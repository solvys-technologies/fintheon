TP's symptom (2026-04-18): sent messages don't return a response, there's a hollow "thinking bubble" that appears before streaming starts, and no streaming happens. Root causes (already diagnosed):

- **Hollow bubble** — [ChatPage.tsx:216-223](mobile/components/chat/ChatPage.tsx:216) pre-creates an assistant message with `content: ""` on send. [ChatMessage.tsx:53-60](mobile/components/chat/ChatMessage.tsx:53) always renders the bubble chrome regardless of content, so an empty message renders as an empty bubble with border. TP sees this as the "blank thinking bubble."
- **Fix**: delay the assistant message insert until the **first `text-delta` event** actually arrives inside the SSE parse loop at [ChatPage.tsx:292](mobile/components/chat/ChatPage.tsx:292). If no event arrives, no empty bubble. You'll need to store `assistantId` locally and conditionally append on first delta. Also handle the case where an `error` or `finish(finishReason=error)` event arrives first — in that case, create a minimal assistant message with the error text (as the existing error-handling branch does).
- **Thinking indicator** in [ChatPage.tsx:568-575](mobile/components/chat/ChatPage.tsx:568) gates on `isLoading && messages[last].content === ""`. After the fix, the last message won't be empty (because we don't insert it yet), so update the gate to `isLoading && (noAssistantYet || lastAssistantEmpty)` where `noAssistantYet` is a boolean you track in local state alongside `assistantId`.
- **Delivery confirmation** — no visual feedback on whether a message hit the backend. Extend `ChatMessageData` at [ChatMessage.tsx:10-15](mobile/components/chat/ChatMessage.tsx:10) with `status?: "sending" | "sent" | "error"`. At [ChatPage.tsx:213](mobile/components/chat/ChatPage.tsx:213), set the user message to `status: "sending"`. On `res.ok` at [ChatPage.tsx:251](mobile/components/chat/ChatPage.tsx:251), flip to `"sent"`. On the error paths, flip to `"error"`. Render a small right-aligned label under the user bubble in [ChatMessage.tsx](mobile/components/chat/ChatMessage.tsx) using `var(--font-data)`, 8px uppercase, matching the `FROM DESKTOP` badge style at [ChatPage.tsx:451-464](mobile/components/chat/ChatPage.tsx:451). Use plain text labels — `…` / `SENT` / `FAILED`. **No emoji, no checkmark glyph, no sparkle.**
- **No-stream watchdog** — add a 12-second watchdog inside `sendMessage`: if zero events arrive within 12s of `res.ok`, surface a non-blocking hint under the user bubble: `HARPER SILENT — CHECK DESKTOP RELAY`. The stream stays open so late events still paint. ### 2. ChatInput caret rendering in the wrong spot ([mobile/components/chat/ChatInput.tsx](mobile/components/chat/ChatInput.tsx))

TP screenshot on 2026-04-18 shows the caret floating in the top-left of the input bar, detached from the placeholder text. The textarea at [ChatInput.tsx:150-182](mobile/components/chat/ChatInput.tsx:150) has:

- `padding: "14px 18px 6px"`
- `lineHeight: 1.45`
- `fontSize: 16`
- `rows={1}` with `handleInput` dynamically resizing to scrollHeight

Likely causes: (a) the textarea's content-alignment isn't respecting the padding-top on initial focus, so the caret renders at content-box origin while the placeholder renders lower; (b) `handleInput` returns a `scrollHeight` on empty content that's smaller than the padded content area, mis-measuring the first frame.

Debug + fix:

- Set `verticalAlign: "top"` and `boxSizing: "border-box"` on the textarea.
- Explicitly set `minHeight` matching one line of padded content (`calc(14px + 6px + ${16 * 1.45}px)` = roughly 43px).
- On mount, run one `handleInput()` synchronously so the textarea's initial height is measured before first focus.
- If the caret still misaligns, try `writingMode: "horizontal-tb"` explicit + `textAlign: "start"` explicit.
- Set `caretColor: "var(--accent, #c79f4a)"` explicit so it doesn't inherit something unexpected.

Verify on-device after the fix by running the mobile build + opening in Safari.
